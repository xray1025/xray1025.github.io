Redis

NoSQL = Not Only SQL

关系型数据库：表格

非关系型数据库：格式不固定，易于横向扩展

传统数据库

  • 结构化组织
  • SQL
  • 数据和关系存在单独的表中
  • 数据定义语言
  • 严格的一致性
  • 基础的事务
  • 。。。

NoSQL

  • 不仅仅是数据
  • 没有固定的查询语言
  • 键值对存储、列存储、文档存储、图形数据库
  • 最终一致性
  • CAP/BASE
  • 高性能、高可用、高拓展
  • 。。。

概述

Redis（remote dictionary server）远程字典服务

  • C
  • 支持网络
  • 内存持久化
  • kv键值对
  • 主从同步
  • 数据库
  • 缓存
  • 中间件

测试工具

redis-benchmark

  • -c指定并发连接数
  • -n指定请求数

基础知识

  • 默认16个数据库

  • 默认使用第0个

    • select 0

    • dbsize

    • keys *

    • 清空当前数据库：flushdb

    • 清空所有：flushall

redis是单线程的

  • redis是基于内存操作的
  • 对内存来说，没有多线程的上下文切换效率更高
  • redis的性能瓶颈不是cpu，而是机器内存和网络宽带
  • 6.0 后是多线程的

数据类型

String（字符串）

  • setex()：设置过期时间（set with exprie）

  • setnx()：不存在则设置（set if not exist）

  • set user{name:alice, age:11}

  • mget/mset

  • getset

List（列表）

  • 所有命令L开头：LPUSH/LRANGE
  • RPUSH/RPOP

Set（集合）

  • 不能重复
  • 所有命令S开头

Hash

  • h开头

Zset

  • 在set的基础上增加了一个值
  • zset k1 score1 v1
  • 以score排序

geospatial

  • geoadd/geodist/geohash/geopos/georadious/georadiousbymember

hyperloglog

  • 基数：不重复的元素，可以接受误差

  • 网页UV：一个人访问网页多次仍算一次访问

bitmap

  • 位运算
  • 统计用户信息 0、1

事务

  • 一组命令的集合

  • 单条命令保证原子性

  • 事务不保证原子性

  • 没有隔离级别的概念

  • 步骤：

    • 开启事务（）：multi
    • 命令入队（）：
    • 执行事务（）：exec

  • 编译时异常，所有命令都不执行

  • 运行时异常，其他命令继续执行

乐观锁

  • 悲观锁：所有行为都加锁

  • 乐观锁：所有行为都不加锁，更新数据的时候获取版本判断数据是否被tar修改

  • redis实现：

    • 使用watch监视
    • 如果执行失败，unwatch解锁
    • 重新监视

Jedis

  • redis官方推荐的java连接工具

SpringBoot整合

  • 在springboot2.x之后，jedis被替换为了lettuce

  • jedis：直连，多线程不安全，需要使用jedis pool连接池 BIO

  • lettuce：netty，多线程共享，可以减少线程数量 NIO

  • 自定义redisTemplate

conf详解

  • unit单位大小写不敏感

  • 包含其他配置文件

  • 网络：

    • bind 127.0.0.1
    • protected-mode yes
    • port 6379

  • daemoize yes # 以守护进程的方式运行，默认为no

  • pidfile /var/run/redis_6379.pid # 制定一个pid文件后台运行

  • loglevel notice # 日志

  • datebase 16

  • snapshot

    • 执行多少次操作则持久化到文件.rdb.aof
    • save 900 1 900s内，至少1个key修改则持久化
    • save 300 10 300s内，至少10个key修改则持久化
    • save 60 100 60s内，至少100个key修改则持久化

  • stop-writes-on-bgsave-error 持久化出错是否继续工作

  • rdbcompression 是否压缩rdb文件

  • rdbchecksum 保存rdb文件是否进行校验

  • dir./ rdb文件保存目录

  • REPLICATION 复制

  • SECURITY 安全 密码默认为空

  • maxclients

  • maxmemory

  • maxmemory-policy noeviction内存满时的处理策略

  • append only # aof配置

    • appendonly no 默认关闭，使用rdb持久化
    • appendfilename
    • appendfsync always/everysec/no

持久化

  • 内存型数据库

RDB

redis database

  • 在指定的时间间隔内将内存中的数据集快照写入硬盘
  • 单独创建一个子进程，先将数据写入临时文件，等待持久化过程结束，再用这个临时文件替换上次持久化好的文件
  • 主进程不进行任何IO操作
  • 适合大规模的数据恢复
  • 最后一次持久化的数据肯丢失
  • 默认RDB
  • dump.rdb

AOF

append only file

  • 将所有命令都记录下来（读操作不记录）

  • 恢复时把文件重新构建

  • 不适合大规模数据恢复

  • redis-check-aof –fix 修复AOF

  • aof文件大于64M则新建AOF

订阅

  • 发送消息：PUBLISH XXX
  • 接收信息：SUBSCRIBE XXX

主从复制

  • 数据的复制是单向的
  • 读写分离
  • 作用
    • 数据冗余
    • 故障恢复
    • 负载均衡
    • 高可用
  • info replication：查看复制信息
  • 修改port、pid、log、dump.rdb

哨兵（sentinel）

  • 主机故障时，从机根据投票自动变为主机

  • 主观下线：一个哨兵发现主机不可用

  • 客观下线：多个哨兵发现主机不可用，哨兵对从机进行投票切换主机

  • 步骤：

    • 配置哨兵：sentinel monitor myredis 127.0.0.1 6379 1
    • 启动哨兵：redis-sentinel

  • 主机重新上线后变为从机

  • 优点

    • 基于主从复制
    • 主从可以切换，可用性高
    • 手动到自动，健壮性好

  • 缺点

    • 难以在线扩容
    • 实现哨兵的配置麻烦

缓存穿透和雪崩、击穿

缓存穿透

  • 大量用户查询的数据在redis内存数据库中没有，继而大量访问持久层数据库
    • 布隆过滤器：对多有可能查询的参数以hash形式存储，在控制层进行校验，不符合则丢弃，从而减小了对底层存储系统的压力
    • 缓存空对象：

缓存击穿

  • 大量请求并发地对一个热点数据进行访问，当这个数据失效的书剑，持续的大并发就穿透缓存，直接访问数据库
    • 设置热点永不过期
    • 加互斥锁

缓存雪崩

  • 在一个时间段，缓存集中过期失效，redis宕机
    • redis高可用
    • 限量降级
    • 数据预热
